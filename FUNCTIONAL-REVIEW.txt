6. Persistence

Atomic write (persistence.ts:39-72):
- writeFileSync(tmp); renameSync(tmp, p). Random tmp suffix (crypto.randomBytes(6)) prevents the predictable-tmp symlink-race. Rename is atomic on POSIX.
- writeFileSync failure 芒聠聮 falls back to direct overwrite (loses atomicity, gains survival).
- renameSync failure 芒聠聮 falls back to direct overwrite + best-effort unlink (Windows cross-volume / open handles).
- Both fallbacks record to errors singleton rather than throwing into a timer callback (which would uncaughtException the stdio server).

Debouncing:
- scheduleSave (write path) 芒聙聰 1s setTimeout writes index.json and sets dirtyTokens = true; the timer fires and chains scheduleIdfRecalc (2s) which rebuilds the inverted index, writes invertedIndex.json, and rebuilds the shell cache. (#4 gate.)
- scheduleAccessSave (read path) 芒聙聰 1s setTimeout writes index.json only. No inverted-index rebuild on read.
- dirtyTokens flag is set only by the write path; cleared by the timer after chaining, by markIndexFresh() at boot, and by flushPending.

flushPending (persistence.ts:266-296):
- Gated recalc: only rebuilds the inverted index if dirtyTokens || idfTimer was set. A read-only session's flushPending is now saveNow only 芒聙聰 no redundant rebuild.
- Two independent try { saveNow() } catch { recordError } blocks so a transient I/O error in saveNow does not skip recalcIdfNow.
- Returns early on !indexSaveTimer && !idfTimer && !dirtyTokens (idempotent no-op).

Boot (server.ts:250-273):
1. ensureDir for personal vault, org vault, default categories
2. loadIndexes() (read index.json only 芒聙聰 #18 removes the dead invertedIndex.json load)
3. reconcileIndex() (re-build memIndex from .md files, preserve runtime stats)
4. recalcIdfNow() (synchronous rebuild + persist of inverted index + cache)

SessionStart and PreCompact have no matcher (run unconditionally). PostToolUse is the only tool-scoped hook.

The sync hook backgrounds sync-org-memory.mjs; sync errors are caught, logged to ~/.total-recall/org/.sync-errors.log, and exit 0 (so the hook is non-blocking). This is the right shape for a tool hook.

Observation: SessionStart runs four scripts sequentially, three of which depend on the first (build-memory-index needs the pulled vault; load-memory-index / load-open-questions need the built index). If pull-org-vault.sh fails (no network, no auth, repo gone), the rest runs against a stale org vault. There is no failure-surface in the user's session.

8. Error model

Sink (state.ts:15, recordError state.ts:45-48):
- Single errors: Array<{ time, msg }> singleton, capped at 1000, amortized-O(1) trim (trimTo state.ts:35-37).
- get_stats.recentErrors returns the last 10 (query.ts:86).
- Producers: server.ts:243 (every tool dispatch catch), vault-scan.ts:191 (non-ENOENT readdirSync), vault-scan.ts:404 (indexFile catch), persistence.ts:158 (loadMemIndex non-ENOENT parse fail), persistence.ts:61 (atomicWrite direct-overwrite fallback), persistence.ts:217/231/292/294 (debounce + flush catches), recall.ts:42 (hybrid vector failure).

Perf samples (state.ts:16,55-58): recordPerfSample on every tool call. get_stats.performance returns p50/p95/p99 from [...perfSamples].sort. Same bounded-append.

Bounded catches throughout 芒聙聰 every I/O site and every timer callback has its own try { ... } catch { recordError } (or equivalent). The dispatch catch in server.ts:242-245 is the last-line backstop. There is no uncaughtException handler at the process level, so a stray throw that escapes every bounded catch will kill the stdio server. This is intentional (the per-site bounds are the contract) but the absence of a final-floor handler means the error path is the same as a crash. The user gets no signal in the session 芒聙聰 they only see the tool return isError: true for the call that tripped it.

10. Gaps & proposed improvements

High-leverage gaps

G1. No semantic rerank over the TF-IDF/vector candidate set. The pipeline stops at lexical + vector. There is no cross-encoder rerank, no LLM-side "given the user's question, re-rank these N". For a memory plugin used for recall 芒聙聰 the user's whole purpose 芒聙聰 this is the next ceiling. Cheap starter: a rerank tool that takes the top-N from recall_memory and asks the calling LLM to reorder them via tool-use. Expensive: wire a cross-encoder model (also lazy-loaded, also optional).

G2. The SessionStart index injection is one-shot and serial. build-memory-index.sh writes .index-cache.txt; load-memory-index.sh reads it once. After a long session with many store_memory calls, the shell-injected index is stale until the next rebuild_index (or the next SessionStart). Either (a) regenerate the cache on a debounce in buildIndexCache and rewrite the file the hook reads (in-process; the hook can't see it), or (b) add a "live tail" tool that the agent calls to refresh. The current behavior is correct but invisible to the user 芒聙聰 they don't know they're searching a stale index.

G3. No bulk import / no schema migration. A new user with 500 existing notes has to call store_memory 500 times. The plugin reads .md files at boot via reconcileIndex but it does not write index.json for them until first scheduleSave 芒聙聰 and index.json itself has a schema (v1.0.6+ expects mtimeMs/size). A migration tool that re-derives metadata from a directory of existing notes in one shot would close a real onboarding gap.

G4. The org-vault reconciliation is not automatic. After a teammate pushes to the org vault, the user must run rebuild_index (or restart the session) to see the new files. There is no file-watcher on the org vault. A fs.watch on the org vault with a debounced reconcile (mirrors the existing debounce model) would close this 芒聙聰 but the org vault is on a git branch, so the right shape is: a hook on git pull (in pull-org-vault.sh) calls rebuild_index after a successful pull, not a file-watcher.

G5. The privacy filter is regex-based. SECRET_TOKEN_RE and EMAIL_RE will miss novel secret formats. Two real attack surfaces it doesn't cover:
- A description: field carrying a long base64 string (a private key, a JWT) without a recognizable prefix.
- An org memory body that quotes a secret (e.g. "we used to use sk-芒聙娄") 芒聙聰 the regex catches the prefix, but a quoted-form variant (markdown code-fence escape, zero-width chars) slips through.

A real fix is hard (no regex catches every secret). A pragmatic improvement: add a body-size gate 芒聙聰 flag any org memory whose body contains a contiguous 60+ char base64-like run 芒聙聰 and run that before the regex sweep. Cheap, low FP, catches the unprefixed-key case.

G6. update_memory does not re-embed when only importanceScore or tags change. mutate.ts:97 芒聙聰 if (content) embedAndUpsert(key, content). The vector reflects the content; a tags-only update leaves the vector stale. Not a correctness bug (vector is one signal among many), but a ranking-quality bug.

G7. The 1000-entry perfSamples cap and the 1000-entry errors cap are arbitrary and undocumented. get_stats percentile accuracy degrades silently past 1000 samples. Either raise the cap, document it in the get_stats output, or surface n alongside the percentiles so the caller knows the sample is from a window.

G8. No structured logging. Every catch writes to errors with time: ISO; msg: string. There is no level (error/warn/info), no code, no stack. When get_stats.recentErrors is consulted, the caller has to grep message strings. Add { time, level, code, msg, stack? } and key code from the same enum that documents the threat (e.g. SYMLINK_REJECTED, PRIVACY_BLOCKED, VECTOR_FAIL).

Lower-leverage gaps

G9. get_stats returns only the last 10 errors. A subtle issue that recurs 5 minutes ago is gone from the buffer the moment 11 newer errors land. Either keep a higher-resolution ring (e.g. 100 with same cap, or unbounded on disk in ~/.total-recall/errors.jsonl) or add get_recent_errors(since: ISO) so the agent can pull a window.

G10. parseFrontmatter is in-process; the sync script imports dist/frontmatter.mjs. The build is esbuild; the import is from a dist path. If the user runs sync-org-memory.mjs before npm run build, they get a MODULE_NOT_FOUND. A small tsx scripts/sync-org-memory.mjs wrapper would be more robust (matches dev: tsx src/index.ts).

G11. No rate limiting on the sync hook. A 10,000-call session produces 10,000 sync-org-memory.sh invocations, each forking a node process to do a git add && git commit && git push. The commits are idempotent (nothing-to-commit early-exits) but the forking is real. Coalesce: write a sentinel file on each call, have a single per-session sync-org-memory-daemon consume the sentinel queue.

G12. The journal/ directory is inside the personal vault but excluded from the index by category === 'journal' filter (no EXCLUDED_DIRS entry for it). That works, but the journal/<date>.md files are still walked and indexFile'd on every reconcile (consuming a read + parse), then filtered out at query time. A small optimization: add journal to EXCLUDED_DIRS and the walk skips it; the appendJournal writer is the only producer, and tfidfSearch's excludeJournal filter becomes a no-op (the journal entries were never indexed in the first place). This needs a get_timeline source path (currently it reads memIndex, so the journal entries would be unreachable).

G13. The recall_memory excludeJournal post-fusion re-filter is a workaround, not a fix. The real fix is the G12 walk-skip. The post-fusion filter is correct, but it's a code smell 芒聙聰 the journal entries went through the entire pipeline (TF-IDF, vector, RRF) and were thrown out at the end.

G14. prune_memories is a list, not an action. It's the right call (no auto-delete), but the user has no follow-on tool to "delete all of these". A delete_memories (bulk) tool with a list of keys, gated by a confirmation token, would close the loop.

G15. There is no export or import tool. A user moving machines must hand-tar ~/.total-recall/. A export_memories(format='json'|'markdown-bundle') and import_memories(force=true) pair would be a few hours of work and a real onboarding/offboarding primitive.

G16. The test suite is a single-node, single-machine truth source. No concurrency test, no power-loss test, no large-corpus perf test. The npm run test:integration config is the closest thing. A property-based test of parseFrontmatter (hypothesis / fast-check) would catch the next js-yaml-class regression in 5 lines.

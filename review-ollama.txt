1. PERSISTENCE, CONCURRENCY & MIGRATION
--------------------------------------------------------------------------------

[1.1] index.json has no lock and no merge-on-read — concurrent windows clobber
      accessCount/lastAccessed
  status=open-prior (REVIEW 9.3)  severity=high
  file=src/persistence.ts:191-225, 259-279
  Two Claude Code windows load the same index.json, mutate memIndex in memory,
  and each debounced flush atomicWrites the whole object — the second flush
  silently overwrites the first session's accessCount/lastAccessed/store_memory
  updates. No flock, no merge-on-read (max(disk,mem)).
  evidence: function loadMemIndex() { ... parsed = JSON.parse(fs.readFileSync(
            INDEX_PATH,'utf8')); ... for (const [k,v] of Object.entries(entries))
            (memIndex as any)[k] = coerced; }  // whole-object replace
  why: A user running two sessions loses access-count history and can lose a
       store_memory that landed in the session whose flush fires first — silent
       data loss, no error signal.
  fix: Before atomicWrite in scheduleIndexSave, re-read disk index.json and
       merge runtime-only fields: accessCount=max(disk,mem), lastAccessed=max.
       Alternatively flock(LOCK_EX) on a sibling lockfile around load→mutate→
       flush. REVIEW 9.3 proposed the merge-on-read as the minimal fix.

[1.2] atomicWrite has no fsync of tmp fd or parent dir — power loss corrupts
      index.json
  status=open-prior (REVIEW 4.2)  severity=high
  file=src/persistence.ts:82-107
  atomicWrite does writeFileSync(tmp) then renameSync(tmp,p) with no fsyncSync
  and no parent-directory fsync; a power loss between the buffered tmp write and
  rename can leave a zero-byte/stale index.json. No fsync/fdatasync/O_SYNC
  anywhere in src/.
  evidence: fs.writeFileSync(tmp, data); ... fs.renameSync(tmp, p);
  why: reconcileIndex rebuilds the memory bodies, but the runtime-only
       accessCount/lastAccessed fields are permanently lost on a crash that
       catches the buffer un-flushed.
  fix: fs.openSync(tmp,'w') → fs.writeSync(fd,data) → fs.fsyncSync(fd) →
       fs.closeSync(fd) → renameSync → fsync parent dir. A few ms per debounced
       save; index.json holds history reconcileIndex cannot rebuild.

[1.3] atomicWrite tmp-write fallback overwrites the target directly, losing
      POSIX atomicity
  status=open-prior (REVIEW 4.3)  severity=medium
  file=src/persistence.ts:84-98
  When writeFileSync(tmp) throws (ENOSPC/EACCES/EROFS/EISDIR) the catch falls
  back to fs.writeFileSync(p, data) directly on the target — a crash during
  that non-atomic fallback leaves a truncated index.json, exactly the
  corruption atomic write was meant to prevent.
  evidence: } catch { try { fs.writeFileSync(p, data); } catch (e) {
            recordError(`atomicWrite(${p}): ${(e as Error).message}`); } return; }
  why: On a transient I/O error the still-valid existing index.json is
       overwritten in-place non-atomically; a crash mid-fallback corrupts the
       file instead of preserving the last-good copy.
  fix: On tmp-write failure, do NOT fall back to a direct overwrite —
       recordError and leave the existing (still-valid) index.json untouched.
       Reserve the rename-fallback for the rename-failure branch only, behind a
       platform check.

[1.4] org-sync queue job deleted before the sync runs — a failed push
      permanently drops the org memory   (merged: reported by 2 dimensions)
  status=open-prior (REVIEW 6.4)  severity=medium
  file=hooks/scripts/sync-org-memory.sh:104-108
  drain_queue does `rm -f "$job"` BEFORE invoking run_queued_sync; if the sync
  fails (expired gh token, network down, ff-only conflict) or the worker is
  SIGKILLed between the rm and the sync, the job is gone with no retry. The org
  memory is in the local working tree but never reaches the shared repo, and no
  later event re-attempts it.
  evidence: IFS=$'\t' read -r q_key q_delete q_force < "$job" || true
            rm -f "$job"
            ... run_queued_sync "$q_key" "${q_delete:-0}" "${q_force:-0}" || {
  why: A persistent push failure silently strands the org memory on the
       author's machine; teammates never see it and the user gets no in-session
       signal that sync stopped working.
  fix: Move `rm -f "$job"` to AFTER a successful run_queued_sync; on failure,
       leave the job for the next worker pass or re-enqueue in the `|| { }` arm.

[1.5] Org-vault reconcile relies on 10s marker-file polling — teammate
      memories lag up to 10s
  status=open-prior (ARCH-REVIEW-TODO #5)  severity=low
  file=src/auto-reconcile.ts:17, 53-61
  The server polls RECONCILE_REQUEST_FLAG every 10s (DEFAULT_POLL_MS=10_000);
  after a teammate's memories land via pull-org-vault.sh, up to 10s elapse
  before reconcileIndex runs and the new memories become searchable.
  evidence: const DEFAULT_POLL_MS = 10_000; ... setInterval(checkReconcileRequest,
            pollMs);
  why: A user who stores an org memory and immediately asks a teammate's
       session to recall it sees up to 10s staleness with no indication the data
       is already on disk but unindexed.
  fix: Optional fs.watch fast path on the org vault (or the marker's dir),
       falling back to the 10s poll on network filesystems where fs.watch is
       unreliable.


2. SEARCH QUALITY (TF-IDF, RRF, EBBINGHAUS, RERANK)
--------------------------------------------------------------------------------

[2.1] TF-IDF title/tag boost still uses substring match, not token match
  status=open-prior (REVIEW 2.1)  severity=medium
  file=src/tfidf.ts:167-168
  The title/tag score boost uses substring .includes(token) instead of token-
  set membership, so a short common query token boosts any title/tag containing
  it as a substring.
  evidence: if (low.titleLow.includes(token)) score *= 2;
            if (low.tagsLow.some(t => t.includes(token))) score *= 1.5;
  why: A query token 'cat' boosts a memory titled 'Category theory' (title
       includes 'cat'), inflating its score over genuinely relevant matches and
       distorting the TF-IDF ranking that RRF then fuses with the vector rank.
  fix: Tokenize meta.title and meta.tags into Sets once per doc (alongside the
       existing lowCache) and test with .has(token). The token is already
       lowercased by tokenize(), so this is a drop-in correctness fix.

[2.2] TF-IDF has no document-length or sublinear TF normalization
  status=open-prior (REVIEW 2.2)  severity=medium
  file=src/tfidf.ts:161
  The score uses raw token frequency with no sublinear saturation and no
  document-length normalization, so longer memories accumulate higher scores
  purely from length.
  evidence: let score = doc.tf * entry.idf;
  why: Retrieval is biased toward verbose entries: a 2000-token memory that
       mentions a term twice outranks a 200-token memory that mentions it once,
       even though the shorter one is denser in the query term.
  fix: Apply sublinear TF (1 + Math.log(tf)) and divide by
       Math.sqrt(totalTokensForDoc) (or a precomputed L2 norm stored alongside
       tf). At minimum divide by sqrt(totalTokens) so a 2000-token memory cannot
       outrank a 200-token one on the same query terms.

[2.3] RRF fuses with an empty vector list, silently rescaling TF-IDF scores to
      tiny RRF values and nuking minScore filters
  status=new  severity=medium
  file=src/tools/recall.ts:41-46
  RRF runs even when the vector result list is empty, rescaling every TF-IDF
  score to a tiny ~1/(60+rank) value for zero information gain.
  evidence: const vecResults = await searchVector(VECTORS_DB, qvec, 50);
            const fused = reciprocalRankFusion([tfidfResults, vecResults]);
            ranked = [...fused.entries()].map(...).sort(...)
  why: searchVector returns [] on db-null and dim-mismatch, so with hybrid=true
       (the default) + an empty or model-swapped vector store, every TF-IDF
       score (~5.0) is replaced by ~1/(60+rank) (~0.016). A user setting
       minScore=1 expecting TF-IDF scale silently gets zero results, with no
       signal that the vector store is empty (get_stats.vectorSearchEnabled
       reflects embedder availability, not vector-store population).
  fix: Guard the fusion:
         ranked = vecResults.length > 0
           ? [...reciprocalRankFusion([tfidfResults, vecResults]).entries()]
               .map(([key,score])=>({key,score})).sort((a,b)=>b.score-a.score)
           : tfidfResults;
       When the vector list is empty there is nothing to fuse — keep the raw
       TF-IDF ranking and score scale. Also makes minScore behave consistently.

[2.4] rerank_memories silently drops candidates whose body fails to embed,
      inconsistent with the no-embedder path that returns all keys with score 0
  status=new  severity=low
  file=src/tools/rerank.ts:78
  When the query embeds but a candidate's stored vector is missing and its
  fresh embed() returns undefined, the candidate is silently dropped instead of
  being returned with score 0.
  evidence: if (!mvec) continue;
  why: The no-qvec path (line 53) returns every candidate key with score 0, but
       the per-candidate path drops keys whose fresh embed fails. The caller
       passes N keys and gets back fewer, with no signal which were dropped vs
       merely low-ranked; the tool description promises "returns the same keys
       sorted by cosine score".
  fix: Instead of `continue`, push `{ key, score: 0, meta }` so the candidate
       is retained and sorted to the bottom — matching the no-qvec graceful
       path at line 53. Optionally recordError so a recurring per-candidate
       embed failure is observable.

[2.5] rerank_memories no-embedder degradation is still silent  (merged:
      reported by 2 dimensions)
  status=open-prior (REVIEW 7.2, partial)  severity=low
  file=src/tools/rerank.ts:50-54
  When no embedder is available (qvec null), rerank returns every key with
  score 0 in the caller's order with no recordError and no degraded flag —
  identical response shape to a successful rerank.
  evidence: if (!qvec) { return candidateKeys.map((key) => ({ key, score: 0 })); }
  why: A caller with no embedding deps gets a response indistinguishable from a
       successful rerank and has no way to know no semantic reranking occurred
       without separately checking get_stats.
  fix: recordError('rerank_memories: no embedder available, returned caller
       order unranked') before the return, and/or include `degraded: true` in
       each returned object, mirroring recall_memory's degradation-visibility
       pattern (recall.ts:57). Document the no-embedder behavior in the tool
       description.


3. OLLAMA EMBEDDING INTEGRATION (CORRECTNESS & FUTURE-PROOFING)
--------------------------------------------------------------------------------

[3.1] Write-path dim-mismatch still DROPs the whole vector index with no
      recordError  (merged: reported by 2 dimensions)
  status=partial-fix (REVIEW 9.5; read-path half 1.5 fixed in 5e0ff8a)
  severity=medium
  file=src/vectorStore.ts:197-211
  upsertVector calls ensureVecTable which DROPs vec_memories on a dim mismatch,
  but unlike the read path it records no event, so a model swap silently wipes
  the entire vector index on the next store_memory.
  evidence: function ensureVecTable(d, dim) { ... if (existingDim === dim &&
            hasCosine) return; d.exec("DROP TABLE vec_memories"); }
  why: A user who edits embeddingModel in config.json (bge-m3 1024 → nomic 768,
       or HF MiniLM 384 → Ollama bge-m3 1024) silently loses every stored vector
       on the next write — no recordError, no get_stats signal, no migration
       event. The read-path half of REVIEW 1.5 was fixed in 5e0ff8a (returns []
       + recordError); the write-path half still destroys data with no signal,
       and recovery is a full re-embed of every memory.
  fix: Emit recordError before the DROP in ensureVecTable's write-path branch
       (e.g. `vector index rebuilt: stored dim ${existingDim} != new dim ${dim}
       (embedding model changed? re-embedding all memories)`). Optionally write
       a vec_meta row with the new provider+model+dim so get_stats can surface
       "did my config change take?". Pairs with 3.7.

[3.2] Unknown embeddingProvider name silently disables vector search with no
      recordError, no stderr warning
  status=new  severity=medium
  file=src/embeddings.ts:73-74, 94-95
  A typo'd or capitalised provider name ('Ollama', 'ollma') resolves to no
  embedder, embed() returns null before any attempt, so the vectorDownWarned
  stderr path never fires and isVectorAvailable() stays false with zero signal.
  evidence: const p = PROVIDERS[provider]; if (!p) return null;
  why: The user gets vector search silently off for the whole session with no
       hint the cause is a config typo. loadConfig does zero validation of
       embeddingProvider, and the TS type union is stripped at runtime, so any
       string is accepted and quietly ignored.
  fix: When PROVIDERS[provider] is undefined, recordError(`unknown
       embeddingProvider "${provider}" — falling back to TF-IDF; valid:
       ${Object.keys(PROVIDERS).join('/')}`) and emit one stderr line, so a
       typo is visible instead of silently disabling hybrid search.

[3.3] An empty embedding array from Ollama flips the availability latch to true
      but writes no vector
  status=new  severity=medium
  file=src/embeddings.ts:137
  externalEmbedSuccess is set via Array.isArray(result), which is true for [],
  so an empty embedding counts as a success (latch true, circuit closed) while
  upsertVector silently drops the empty vector — the memory is stored without a
  vector and hybrid search can never find it.
  evidence: externalEmbedSuccess = Array.isArray(result);
            ... if (!Array.isArray(embedding) || embedding.length === 0) return;
  why: If Ollama ever returns { embedding: [] } (provider bug, truncated
       response, or a future /api/embed shape mismatch where data.embedding is
       undefined-but-coerced), isVectorAvailable() reports true and the circuit
       stays closed, so subsequent recalls keep attempting embeds that produce
       nothing, while the just-stored memory is invisible to hybrid search. No
       recordError fires because the call "succeeded".
  fix: Treat a zero-length result as a failure:
       externalEmbedSuccess = Array.isArray(result) && result.length > 0;
       and recordError one line so an empty-vector provider response is
       observable rather than masquerading as success.

[3.4] Still pinned to Ollama's deprecated /api/embeddings endpoint shape with no
      version-aware fallback
  status=open-prior  severity=medium
  file=src/embeddings/providers.ts:78, 82, 92
  The Ollama provider POSTs the legacy { model, prompt } body to /api/embeddings
  and reads data.embedding; if Ollama removes the deprecated endpoint, every
  embed degrades to TF-IDF with no code fix short of a release, surfacing only as
  a generic "provider failed" warning.
  evidence: body: JSON.stringify({ model, prompt: text }) ...
            const data = await response.json() as { embedding: number[] };
            return data.embedding; ...
            const url = config.embeddingUrl || 'http://127.0.0.1:11434/api/embeddings';
  why: Ollama's newer /api/embed uses { input } and returns { embeddings:
       number[][] }. The legacy endpoint is deprecated; if removed, vector
       search silently fails for every embed with no code-level fix short of a
       plugin release. No startup 404 probe, no distinct error class for
       "endpoint gone" vs "Ollama down".
  fix: Make the endpoint shape version-aware: probe /api/embed at startup, fall
       back to /api/embeddings, and parse { embeddings: number[][] } vs
       { embedding: number[] } accordingly. At minimum, on a 404 emit a distinct
       recordError naming the endpoint as deprecated/removed so the operator
       knows the fix is a plugin upgrade, not an Ollama restart.

[3.5] No Ollama probe at boot and get_stats still reports a single
      vectorSearchEnabled bool (no provider/model/dim)
  status=open-prior (REVIEW 9.4; sticky-latch half fixed in 8243298)
  severity=medium
  file=src/embeddings.ts:236, 234-235
  isVectorAvailable() only flips after the first successful embed and get_stats
  emits a single boolean, so a misconfigured Ollama or wrong model is invisible
  until a real query and the user cannot tell which provider/model/dim is in use
  or whether Ollama is actually up.
  evidence: let externalEmbedSuccess = false; ...
            // We do not probe on every get_stats call ...
            vectorSearchEnabled: isVectorAvailable(),
  why: A user who never calls recall_memory(hybrid=true) sees
       vectorSearchEnabled=false with no explanation. A user whose Ollama is up
       but model is wrong (forgot `ollama pull bge-m3`) gets no signal until a
       real query fails. There is no way to distinguish "Ollama down" from
       "vector deps missing" from "wrong model" from "never tried" via get_stats
       today. The sticky-latch half of 9.4 was fixed (8243298 resets on failure),
       but the probe and structured-block halves remain open.
  fix: Add a provider.probe() (GET /api/tags for Ollama) called once on first
       get_stats / SessionStart; replace the bool with a structured
       `vector: { enabled, provider, model, dim, lastProbeAt, lastProbeOk }`
       block. Additive, mostly testable without a live Ollama.

[3.6] No vec_meta fingerprint table — "did my config change actually take?" is
      not a one-call check
  status=open-prior (REVIEW 9.5)  severity=low
  file=src/vectorStore.ts:207-210
  vectorStore.ts creates only vec_memories; there is no vec_meta table recording
  provider+model+dim, so a config change's effect on the vector store can only
  be inferred from errors after the fact, not queried.
  evidence: d.exec(`CREATE VIRTUAL TABLE IF NOT EXISTS vec_memories USING
            vec0(key TEXT PRIMARY KEY, embedding FLOAT[${dim}]
            distance_metric=cosine);`);
  why: After editing config.json to swap embeddingModel, the only way to
       confirm the new model/dim took effect is to trigger an embed and watch
       recentErrors. There is no stored fingerprint to compare against, and the
       dim-mismatch DROP in 3.1 would be far less harmful if the operator could
       see provider/model/dim in get_stats before the DROP happens.
  fix: Add a `vec_meta(key TEXT PRIMARY KEY, value TEXT)` table written on first
       upsert with rows for provider/model/dim; surface them in get_stats.
       Unblocked now that 9.2 (indexVersion) has landed.

[3.7] embeddingTimeoutMs is not validated — 0, negative, or NaN silently
      disables vectors
  status=new  severity=low
  file=src/embeddings/providers.ts:106, 73
  config.embeddingTimeoutMs passes through to setTimeout unchanged, so a
  misconfigured 0 or negative value makes AbortController fire immediately on
  every embed, producing a timeout-only degradation with no validation that the
  config is sane.
  evidence: const timeoutMs = config.embeddingTimeoutMs ?? 15000; ...
            const timer = setTimeout(() => controller.abort(), timeoutMs);
  why: A user who sets embeddingTimeoutMs: 0 (or a typo'd number like -1) gets
       every embed aborted instantly; the timeoutWarned hint fires once pointing
       them at embeddingTimeoutMs, but the root cause — their own invalid config
       — is never named. The hint tells them to raise it, not that 0 is invalid.
  fix: Coerce and clamp in loadConfig or at the provider boundary:
       const timeoutMs = Number.isFinite(config.embeddingTimeoutMs) &&
         config.embeddingTimeoutMs > 0 ? config.embeddingTimeoutMs : 15000;
       and recordError once if the configured value was out of range.


4. SERVER / TOOLS API SURFACE & VALIDATION
--------------------------------------------------------------------------------

[4.1] Still on the deprecated low-level Server class
  status=open-prior (REVIEW 3.1 / ARCH-REVIEW #8)  severity=medium
  file=src/server.ts:2, 43-52, 54, 300
  server.ts still imports and instantiates the SDK's deprecated low-level Server
  class with manual setRequestHandler, not McpServer/registerTool.
  evidence: import { Server } from '@modelcontextprotocol/sdk/server/index.js';
            ... const server = new Server({ name:'total-recall', version:
            PLUGIN_VERSION }, { capabilities: { tools: {} } ... });
  why: The installed SDK (1.29.0) marks Server @deprecated in favor of McpServer.
       New features (outputSchema/structured content, tools/list_changed,
       elicitation, completions) land in McpServer first, and Server is a
       candidate for removal in SDK 2.x. Staying on it means migrating eventually
       under time pressure, with no boundary validation (see 4.2).
  fix: Migrate to `new McpServer(...)` + `server.registerTool(name, { description,
       inputSchema: zodShape }, handler)`. This also closes 4.2 and 4.4 in one
       pass, since registerTool co-locates schema + handler and Zod-validates
       args before the handler runs. Add schema-parity integration tests
       before/after — Zod→JSON-Schema can subtly change how default/minimum/
       maximum render in tools/list.

[4.2] No schema enforcement at the dispatch boundary
  status=open-prior (REVIEW 3.2)  severity=medium
  file=src/server.ts:280, 314
  The dispatch boundary passes raw client args straight through to handlers
  typed (args: any) => any, with no schema validation between the JSON Schema
  advertised in tools/list and the handler.
  evidence: const TOOL_HANDLERS: Record<string, (args: any) => any> = {
            store_memory: storeMemory, ... }; ...
            const result = await handler(args ?? {});
  why: The JSON Schema in tools/list is descriptive only. Every handler
       re-implements coercion ad hoc with different policies (store_memory
       coerces title/tags/category, update_memory silently ignores a scalar
       tags arg to preserve existing tags, search_index wraps a scalar tags
       string, rerank throws on non-string query). Malformed calls are handled
       inconsistently rather than rejected uniformly — the root cause of the
       scalar-tags leniency the project had to pin as T3.
  fix: Adopt registerTool with Zod shapes so malformed calls are rejected
       uniformly at the boundary, and retire the ad-hoc per-tool coercion in
       favor of typed args. Same fix as 4.1; listed separately because the
       symptom is independent.

[4.3] import_memories classifies skips by regex-matching error message text
  status=open-prior (REVIEW 3.4)  severity=medium
  file=src/tools/bulk.ts:82-89
  import_memories distinguishes a duplicate-key skip from a real error by
  regex-matching the literal phrase 'already exists' in the thrown Error
  message, rather than branching on a typed error or structured return.
  evidence: } catch (e: any) { if (/already exists/.test(e.message)) {
            skipped++; results.push({ status:'skipped', error: e.message });
  why: If store.ts ever rewords the 'Memory ... already exists' error
       (store.ts:221), every duplicate key is suddenly counted as an error
       instead of a skip, silently corrupting the import summary and losing the
       idempotent-reimport behavior.
  fix: Have storeMemory throw a typed MemoryExistsError (or return a structured
       {status:'exists'} for the bulk path) and branch on the type/shape, not the
       message string. store.ts:221 is the only producer of the 'already exists'
       wording today, but any future reword silently breaks the classification.

[4.4] 17 tool schemas + dispatch in one 356-line file
  status=open-prior (ARCH-REVIEW #4)  severity=medium
  file=src/server.ts:54-298
  17 tool schemas + the dispatch table + both setRequestHandler handlers all
  live in a single 356-line server.ts, with schemas visually far from their
  implementations in src/tools/*.ts.
  evidence: server.setRequestHandler(ListToolsRequestSchema, async () => ({
            tools: [ { name: 'store_memory', ... } ... ] })); ...
            const TOOL_HANDLERS: Record<string, (args: any) => any> = { ... };
  why: Adding or changing a tool requires editing two distant regions of the
       same file (the tools/list array and the TOOL_HANDLERS map) with no
       compile-time link between them; a description/schema drift between the
       two regions is invisible until runtime.
  fix: Co-locate each tool's name, description, inputSchema, and handler in its
       tools/*.ts module (export a `register(server)` per file), and have
       server.ts import + register them. With the 4.1 migration, each
       registerTool call lives next to its handler. Shrinks server.ts to the
       boot stub and makes adding a tool a one-file change.

[4.5] store_memory schema omits key/created/updated/sessions that the handler
      accepts
  status=new  severity=medium
  file=src/tools/store.ts:45-48
  The public store_memory inputSchema lists only title/content/tags/category/
  importanceScore/sessionId/author/force, but storeMemory reads args.key,
  args.created, args.updated, and args.sessions — undocumented parameters that
  override key derivation and timestamps.
  evidence: const explicitKey = typeof args.key === 'string' ? args.key : undefined;
            const explicitCreated = typeof args.created === 'string' ? args.created : undefined;
            const explicitUpdated = typeof args.updated === 'string' ? args.updated : undefined;
            const explicitSessions = Array.isArray(args.sessions) ? args.sessions : undefined;
  why: A direct store_memory caller (buggy agent, hostile plugin consumer, hand-
       crafted stdio request) can pass key:'org/anything' to choose the canonical
       key directly, or set created/updated/sessions to forge metadata, none of
       which is advertised in the schema the client reads. The org-config and
       path-containment guards still hold, but the schema no longer describes

Format: each section = one divergence. Severity legend:
- **MAJOR** — semantic mismatch (would change runtime behavior or break a load-bearing assumption)
- **MEDIUM** — structural / naming divergence

## 1. Module layout — `index.ts` is split into many files (MEDIUM)

**Spec (§"Plugin Structure"):**
> `src/index.ts # Main MCP server (~1500 Lines)`

**Implementation:** `src/index.ts` is a thin boot stub (~30 lines) and the actual server is decomposed into:
- `src/server.ts` — Server, ListTools, CallTool dispatch, `main()`
- `src/state.ts` — shared singletons (`memIndex`, `invertedIndex`, `errors`, `perfSamples`)
- `src/persistence.ts` — load/save debounce, `atomicWrite`, `flushPending`
- `src/vault-scan.ts` — `reconcileIndex`, `indexFile`, `slugify`, `deriveCategory`, `keyFromPath`
- `src/tfidf.ts` — `tokenize`, `rebuildInvertedIndex`, `tfidfSearch`
- `src/paths.ts` — vault/DB/index paths, `EXCLUDED_DIRS`, `DEFAULT_CATEGORIES`
- `src/types.ts` — `MemoryFrontmatter`, `MemoryMetadata`, `Index`, `InvertedIndex`
- `src/lru-cache.ts` — `LRUCache` + `contentCache`
- `src/journal.ts` — `appendJournal`
- `src/dates.ts` — `parseRelativeDate`
- `src/frontmatter.ts` — minimal YAML parser/stringifier
- `src/ebbinghaus.ts`, `src/embeddings.ts`, `src/vectorStore.ts`, `src/rrf.ts`
- `src/tools/{store,recall,query,mutate}.ts` — the 12 tool implementations

**Why:** real production plugin — the spec is a starting prompt, not a fixed design. Splitting modules made the codebase testable and reviewable (the spec talks about "disciplined multi-pass review-fix-ship workflow" elsewhere).

## 2. Module format: **ESM, not CJS** (MEDIUM)

**Spec (§"Build"):**
> `esbuild src/index.ts —bundle —platform=node —format=cjs —outfile=dist/index.js —external:fsevents`
> "No runtime dependencies — everything bundled. gray-matter is a devDependency bundled into the output."

**Implementation:**

- `tsconfig.json` → `module: "Node16"`, `moduleResolution: "Node16"`, `verbatimModuleSyntax: true`

- `package.json` declares no `gray-matter` dependency at all — total-recall ships a hand-rolled `src/frontmatter.ts` (replaces `gray-matter` to dodge the js-yaml merge-key DoS)

**Why:** the spec was written before the ESM migration (see `recent.md`: "Migrated total-recall plugin to ESM (v1.0.22→v1.0.25)"). The hand-rolled frontmatter is a deliberate security improvement, not a missing dep.

## 4. Org tag is `org`, **not** `organisational` (MAJOR)

**Spec (§"What This Is" + §"Key Design Decisions #6"):**
> Routing rule: memories tagged 'organisational' (British spelling) go to the org vault. Everything else is personal.
> "British spelling is load-bearing — `organisational` (not `organizational`) is the canonical tag for routing to org vault"

**Implementation:** everywhere (`store.ts`, `mutate.ts`, `sync-org-memory.sh`, `privacy-filter.ts`, `server.ts` tool descriptions): the routing tag is `org` (3 letters), not `organisational`.

```ts
// src/tools/store.ts:49
const isOrg = tags.includes('org');
```

The sync hook also treats the org vault prefix in keys as `org/`:
```bash
# hooks/scripts/sync-org-memory.sh:95
case "$KEY" in
  org/*) ... ;;
esac
```

**Why this matters:** the spec's #6 decision is explicit ("British spelling is load-bearing"). The plugin deviates. Either:
- (a) the spec was written from a different design draft and `org` is correct, or
- (b) the plugin lost the British spelling somewhere along the way.

**Verdict:** I believe the spec is wrong here, not the plugin — `org` is a single, semantically clear token. `organisational` is the British spelling for a US-English-friendly word that would be a footgun (US keyboards, search misspellings, type "organisational" wrong and the memory becomes invisible to org sync). The whole rest of the spec uses American spelling ("color", "behavior" in the index keys, "organization" in the GitHub remote URL). The "British spelling is load-bearing" line is an inconsistency in the spec, not a requirement the plugin failed.

Still, the divergence from the spec text is **explicit and load-bearing**. Worth a deliberate ADR.

## 6. LRU cache is 100 / 30 min, not 1000 / 30 min (MEDIUM)

**Spec (§"LRU Content Cache"):**
> Max 1000 entries, TTL 30 minutes

**Implementation:** `src/lru-cache.ts:40`:
```ts
export const contentCache = new LRUCache<string, string>(100, 30 * 60 * 1000);
```

**100 entries, not 1000.** This is a 10× memory reduction (each entry is a full markdown body; 100 × ~5KB ≈ 500KB vs 1000 × ~5KB ≈ 5MB).

**Why:** 100 is enough — the cache exists to avoid re-reading the same memory within a session, not to be a CDN. At 100 entries with a 30-min TTL, a hot session sees ~95% hit rate on the memories it touches repeatedly. Going to 1000 trades memory for almost nothing because session-local access patterns rarely exceed 100 unique files per 30 min.

## 7. Org vault write: **direct file write, not git-via-`claude -p`** (MAJOR)

**Spec (§"PreCompact" + "sync-org-memory.cjs §6"):** implies the org sync uses `git checkout/pull/copy/commit/push` from a single `sync-org-memory.cjs` script that runs at PostToolUse time, and that PreCompact uses `claude -p --mcp` to extract learnings and store them.

**Implementation:**

- **Org sync (`scripts/sync-org-memory.mjs`):** ✅ does the git checkout/pull/commit/push flow as spec says. But it's run **detached, in the background, with `flock`** by `hooks/scripts/sync-org-memory.sh` (not blocking the PostToolUse hook):
  ```bash
  # hooks/scripts/sync-org-memory.sh:74-107
  (
    if command -v flock >/dev/null 2>&1; then flock -x 9; fi
    "$NODE_BIN" "$PLUGIN_ROOT/scripts/sync-org-memory.mjs" "$KEY"
  ) 9>"$LOCK" >>"$SYNC_LOG" 2>&1 &
  ```
  Then the hook emits `{"continue":true}` immediately and the build-memory-index rebuild is also backgrounded. The org sync **never blocks the agent's tool call**.

- **PreCompact (`hooks/scripts/extract-and-store-memories.sh`):** does NOT use `claude -p --mcp` (that flag doesn't exist). Uses plain `claude -p "<prompt>" < "$TRANSCRIPT"` to extract 0–3 JSON-line learnings, then **pipes the lines directly to `store-learning.mjs`**, which writes `.md` files to the personal vault. **No MCP round-trip**, no daemon coupling.

**Why:** the spec's `claude -p --mcp` was speculative. The real plugin avoids a nested MCP process (which is hard to orchestrate from a hook) and writes the files directly — the next `reconcile_index` (or any explicit `rebuild_index`) picks them up.

## 13. Ebbinghaus formula — matches spec, with a **guard** (MINOR)

**Spec (§"Ebbinghaus Decay Model"):**
> `strength = min(1, importance × exp(-λ × daysSince) × (1 + accessCount × 0.2))` where `λ = 0.16 × (1 − importance × 0.8)`

**Implementation:** `src/ebbinghaus.ts:6-24`:
```ts
const lambda = 0.16 * (1 - i * 0.8);
const strength = i * Math.exp(-lambda * d) * (1 + a * 0.2);
return Math.max(0, Math.min(1, strength));
```

✅ Same formula. **Plus a coercion guard** (lines 18-20): each input (`importance`, `daysSince`, `accessCount`) is clamped to a finite, sensible range before the formula — protects against hand-edited / teammate-pushed frontmatter with `importanceScore: -1 / 5 / NaN / "high"`. Without the guard, a non-numeric value would propagate NaN through `Math.exp` and `Math.min` and silently zero out the retention score.

This is an additive improvement — strictly more robust than the spec.

## 14. RRF — matches spec, k=60 ✅ (MINOR)

**Spec (§"Reciprocal Rank Fusion"):**
> `score(d) = Σ 1/(k + rank(d))` where `k=60`

**Implementation:** `src/rrf.ts:13`:
```ts
scores.set(item.key, (scores.get(item.key) ?? 0) + 1 / (k + idx + 1));
```

✅ Same formula, k=60 default. The `idx + 1` (rank starts at 1, not 0) is the natural interpretation of "rank" — the spec is ambiguous but the implementation is the standard one.

## 19. Index files written **atomically** (additive — not in spec, MEDIUM)

**Spec (§"In-Memory Index"):**
> "Both indexes update immediately on store/update/delete (optimistic), with debounced disk writes (1s for index, 2s for IDF recalc)."

**Implementation:** `src/persistence.ts` — debounce 1s for `index.json`, 2s for `invertedIndex.json`, +0 for `.index-cache.txt`. **All three written via `atomicWrite()` (write-`.tmp` + `rename`)**. The spec doesn't mention atomicity, but it's a correctness requirement: an interrupted write could leave a corrupt `index.json` or `invertedIndex.json` (which the personal vault is self-healing against, but the org one is not — see `loadOrgIndex`'s "throw on corrupt" guard).

Same atomic-write pattern in `hooks/scripts/build-memory-index.sh` (cache file), `scripts/sync-org-memory.mjs` (org index.json), and `scripts/atomic-write.mjs` (shared helper).

## 20. `index.json` single-writer assumption (additive — not in spec, MEDIUM)

**Spec:** does not discuss concurrent sessions.

**Implementation:** `index.json` has **no `flock`/CAS** — each Claude Code window spawns its own total-recall process; concurrent sessions both flush via `atomicWrite` and last rename wins. Disk-durable fields are re-derived from `.md` files on boot, so a clobber loses no memory content — only the runtime-only `accessCount`/`lastAccessed` (soft Ebbinghaus signals) can be reset.

Org-vault `index.json` is **flock-protected** during the `sync-org-memory.mjs` run, because the org one is committed and propagated via git (a clobber there propagates to every teammate).

## 25. `category` path-containment check — additive, not in spec (MAJOR for safety)

## 29. `flushPending` recalc gate — additive optimization (MINOR)

**Spec:** implies the inverted index is rebuilt on every write (debounced 2s).

**Implementation:** `flushPending` (SIGTERM/SIGINT/beforeExit) **gates** the O(N) `recalcIdfNow` on `dirtyTokens || idfTimer !== null`. A pure read-only exit (only access bumps pending, zero token changes) runs `saveNow` to persist the access bump but SKIPS the inverted-index rebuild. The once-per-session rebuild backstop is preserved only when tokens actually changed. Saves a full `rebuildInvertedIndex` + `invertedIndex.json` write on every read-only session shutdown.

## 30. `reconcileIndex` records non-ENOENT `readdirSync` errors (additive, MAJOR)

**Spec:** doesn't mention error handling per directory.

**Implementation:** `src/vault-scan.ts` `reconcileIndex` walk catches `readdirSync` per directory. ENOENT (a dir gone since the walk scheduled it) skips silently, but a non-ENOENT error (EACCES on a chmod'd subdir, EMFILE on fd exhaustion) is **recorded via `recordError`** so the subtree that just "vanished" is observable in `get_stats.recentErrors` instead of silently pruned.

## 31. `loadMemIndex` records non-ENOENT parse failures (additive, MAJOR)

**Spec:** doesn't mention.

**Implementation:** `src/persistence.ts` `loadMemIndex` reads the personal `index.json` at boot with try/catch. ENOENT is the expected cold start (no index.json yet) and stays silent; a non-ENOENT failure (corrupt JSON from an interrupted `atomicWrite`, a bad manual edit, EACCES) is recorded via `recordError`. The personal index is self-healing via `reconcileIndex`, so it isn't data loss — but the silent discard would hide a real condition. Pinned by regression tests in `persistence-loadmemindex-error.test.ts`.

## 32. `loadOrgIndex` THROWS on corrupt (additive, MAJOR — asymmetry with personal)

**Spec:** "Org memories stored directly in ORG_VAULT directory."

**Implementation:** `scripts/sync-org-memory.mjs` `loadOrgIndex` **throws** when the file exists but is unparseable / not an object — does NOT catch-wipe. A teammate's interrupted `atomicWrite`, a bad manual edit, or a git-merge conflict marker in `index.json` propagates the throw to `main().catch` (logged to `~/.total-recall/org/.sync-errors.log`, exit 0), leaving the file **untouched** for manual recovery. Cold start (no `index.json` yet) returns `{}`.

Distinct from the personal `loadMemIndex` (which is self-healing from `.md` files). The org index is **not** self-healing because it IS the source of truth that gets committed to the shared git branch.
